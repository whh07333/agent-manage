## 一、架构设计原则
### 1.1 核心设计理念
- **API优先**：所有功能100%通过API暴露，Agent无需界面操作
- **高可靠性**：事件不丢失、数据不丢失、服务不宕机
- **高性能**：支持1000+Agent并发，平均响应时间≤50ms
- **可扩展性**：支持功能迭代和第三方集成，无需重构核心架构
- **云原生**：容器化部署、弹性扩缩容、可观测性完备

### 1.2 架构选型依据
需求已明确技术栈推荐：Node.js + Express + PostgreSQL + 飞书API
结合高并发、高可靠要求，最终确定架构方案。

### 2.2 各层职责说明
#### 接入层
- **RESTful API**：提供所有CRUD操作接口，遵循RESTful规范，返回JSON格式
- **WebSocket API**：提供实时数据推送，支持前端界面实时更新（延迟≤1秒）
- **HTTP回调推送**：向订阅的Agent推送事件通知，支持重试和死信队列

#### 网关层
- **身份鉴权**：对接OpenClaw身份系统，验证API密钥有效性和权限范围
- **限流熔断**：防止恶意攻击和流量突增，保障系统稳定性
- **日志审计**：全量记录所有请求和响应，支持追溯和审计

#### 业务层
- **项目管理模块**：项目CRUD、配置管理、状态流转、归档管理
- **任务管理模块**：任务CRUD、状态变更、依赖管理、交付物管理
- **事件推送模块**：事件订阅管理、推送调度、重试机制、死信队列管理
- **统计分析模块**：项目进度统计、Agent负载统计、效率指标计算、监控看板数据
- **权限管理模块**：RBAC权限模型实现、数据隔离、敏感字段脱敏
- **审计日志模块**：操作日志记录、查询、回溯、导出
- **密钥管理模块**：对接OpenClaw身份系统，实现密钥颁发、存储、轮换、吊销
- **扩展集成模块**：第三方系统对接、自定义工作流、扩展字段支持

#### 数据层
- **PostgreSQL主库**：存储业务核心数据，支持事务、高并发写入
- **PostgreSQL从库**：负责读请求负载，提高查询性能
- **Redis缓存**：缓存热点数据（项目信息、任务状态、统计结果），降低数据库压力
- **Kafka消息队列**：异步处理事件推送、日志写入、统计计算等非核心路径
- **对象存储OSS**：存储大体积交付物、备份数据、历史归档
- **全文检索引擎**：支持项目、任务、日志的全文检索和高级查询

#### 基础设施层
- **Docker容器**：服务打包、环境一致性、快速部署
- **Kubernetes编排**：弹性扩缩容、服务发现、负载均衡、故障自愈
- **监控告警系统**：指标采集、性能监控、异常告警、链路追踪
- **日志平台**：日志聚合、查询、分析、故障排查
- **配置中心**：配置集中管理、动态更新、环境隔离
- **CI/CD流水线**：自动化构建、测试、部署，保障发布质量

## 三、核心模块架构设计
### 3.1 事件推送系统架构
```
┌─────────────┐     ┌─────────────┐     ┌─────────────┐     ┌─────────────┐
│  状态变更   │────▶│  事件生产   │────▶│  Kafka队列  │────▶│  推送调度   │
└─────────────┘     └─────────────┘     └─────────────┘     └──────┬──────┘
                                                                  │
                    ┌─────────────┐     ┌─────────────┐     ┌──────▼──────┐
                    │  死信队列   │◀────│  重试队列   │◀────│  HTTP推送   │
                    └─────────────┘     └─────────────┘     └─────────────┘
```
**设计要点**：
1. 所有状态变更事件首先写入Kafka，确保不丢失
2. 推送调度器消费队列，异步向订阅Agent推送
3. 推送失败的事件进入重试队列，采用指数退避策略（1min→5min→15min）
4. 连续3次推送失败的事件进入死信队列，保留7天，支持手动重发
5. Agent恢复订阅后自动补推暂停期间的所有事件

### 3.2 API密钥管理架构
```
┌─────────────┐     ┌─────────────┐     ┌─────────────┐
│  Agent注册  │────▶│ OpenClaw身份│────▶│  密钥颁发   │
└─────────────┘     │   系统      │     └─────────────┘
                    └──────┬──────┘
                           │
┌─────────────┐     ┌──────▼──────┐     ┌─────────────┐
│  密钥轮换   │◀────│  密钥存储   │────▶│  权限校验   │
└─────────────┘     │  (加密)     │     └─────────────┘
                    └─────────────┘
```
**设计要点**：
1. 完全对接OpenClaw现有身份系统，无需独立开发密钥管理
2. 密钥加密存储，不可明文导出，仅Agent本人和管理员可见
3. 支持自动/手动轮换，旧密钥保留24小时过渡期
4. 密钥泄露后可立即吊销，旧密钥立即失效
5. 所有密钥操作记录审计日志，可追溯

### 3.3 数据权限架构
```
┌─────────────┐     ┌─────────────┐     ┌─────────────┐
│  请求接入   │────▶│  身份校验   │────▶│  权限匹配   │
└─────────────┘     └─────────────┘     └──────┬──────┘
                                                  │
                    ┌─────────────┐     ┌──────▼──────┐
                    │  脱敏处理   │◀────│  数据查询   │
                    └─────────────┘     └─────────────┘
```
**设计要点**：
1. 采用RBAC权限模型，预设4种角色（系统管理员/项目负责人/普通成员/观察者）
2. 数据完全隔离，Agent仅可访问授权范围内的项目数据
3. 跨项目统计分析需要系统管理员特殊授权
4. 敏感字段（API密钥、隐私信息）自动脱敏，仅保留前3位和后3位
5. 所有数据访问操作记录审计日志

## 四、关键技术方案
### 4.1 高并发处理方案
1. **读写分离**：PostgreSQL主从架构，写请求走主库，读请求走从库
2. **多级缓存**：Redis缓存热点数据，本地缓存静态配置，降低数据库压力
3. **异步处理**：非核心路径（事件推送、日志写入、统计计算）通过Kafka异步处理
4. **限流降级**：网关层实现令牌桶限流，峰值流量下降级非核心功能
5. **连接池优化**：数据库、Redis、HTTP客户端都使用连接池，减少连接开销

### 4.2 数据可靠性方案
1. **数据库多副本**：PostgreSQL三副本部署，RPO=0，RTO≤30秒
2. **定期备份**：每日全量备份+实时增量备份，备份保留30天
3. **事件持久化**：所有事件写入Kafka持久化存储，保留7天
4. **幂等性保障**：所有写接口支持Request ID幂等，重复请求不产生脏数据
5. **数据校验**：写入时校验数据完整性，定期巡检数据一致性

### 4.3 实时推送方案
1. **前端实时更新**：采用SSE（Server-Sent Events）实现服务器推送，延迟≤1秒
2. **Agent事件推送**：HTTP回调推送，支持重试和死信队列，成功率≥99.99%
3. **增量更新**：仅推送变更数据，避免全量同步浪费带宽
4. **断线重连**：客户端自动断线重连，重连后自动补推缺失数据

### 4.4 可观测性方案
1. **指标监控**：采集API调用量、响应时间、错误率、事件推送成功率等核心指标
2. **日志聚合**：全量日志采集到统一平台，支持按Trace ID查询全链路日志
3. **链路追踪**：实现分布式链路追踪，定位跨服务调用的性能瓶颈和故障点
4. **告警体系**：多维度告警规则（错误率突增、响应变慢、队列堆积等），及时通知运维人员

### 5.3 扩容策略
- **API服务**：无状态，根据CPU/内存使用率自动水平扩缩容

- **消息队列**：根据队列堆积情况动态扩容分区和节点
- **缓存**：缓存命中率低于90%时扩容节点或调整缓存策略

## 七、安全架构设计
### 7.1 网络安全
- 所有服务部署在私有VPC，不直接暴露公网
- 公网访问仅通过负载均衡的HTTPS端口
- 数据库、缓存等组件禁止公网访问，仅内网可连接
- 网络策略隔离不同服务的访问权限

### 7.2 应用安全
- API密钥鉴权，所有请求必须携带有效密钥
- 输入参数校验，防止SQL注入、XSS攻击
- 敏感数据加密存储，传输使用HTTPS
- 定期安全漏洞扫描，及时修复高危漏洞

### 7.3 数据安全
- 数据多副本存储，定期备份异地容灾
- 敏感字段自动脱敏，防止数据泄露
- 操作日志留存≥180天，支持审计追溯
- 数据删除遵循软删除策略，可恢复周期≥7天

## 九、风险与应对措施
| 风险项 | 可能性 | 影响程度 | 应对措施 |
|--------|--------|----------|----------|
| 高并发场景性能瓶颈 | 中 | 高 | 提前压测优化，做好限流降级预案 |

| 架构迭代重构风险 | 低 | 中 | 做好接口抽象，预留扩展点，迭代不破坏现有兼容性 |
