## 2. 代码管理规则

### 2.1 Git 使用规范

#### 分支命名规则
- **功能开发：** `feature/功能名称`（如：feature/用户管理）
- **Bug 修复：** `fix/问题描述`（如：fix/登录验证问题）
- **重构：** `refactor/重构范围`（如：refactor/数据处理逻辑）
- **文档更新：** `docs/文档类型`（如：docs/接口文档）
- **发布分支：** `release/版本号`（如：release/v1.0.0）
- **热修复：** `hotfix/问题描述`（如：hotfix/安全漏洞）

#### 提交信息规范

```
<类型>(<范围>): <简短描述>

# 示例
feat(用户): 添加用户注册功能
fix(登录): 修复登录验证失败问题
docs(接口): 更新API接口文档
refactor(数据): 重构数据处理逻辑
test(用户): 添加用户模块单元测试
chore(依赖): 更新项目依赖版本
```

**类型说明：**
- `feat` - 新功能
- `fix` - Bug 修复
- `docs` - 文档更新
- `style` - 代码格式
- `refactor` - 重构
- `test` - 测试相关
- `chore` - 构建/工具相关

#### 提交流程

```bash
# 1. 确保在主分支最新状态
git checkout main
git pull origin main

# 2. 创建功能分支
git checkout -b feature/功能名称

# 3. 编写代码并提交
git add .
git commit -m "feat(功能): 添加XX功能"

# 4. 定期同步主分支
git fetch origin
git rebase origin/main

# 5. 推送到远程分支
git push origin feature/功能名称

# 6. 创建 Pull Request
gh pr create --base main --head feature/功能名称 \
  --title "feat: 添加XX功能" \
  --body "实现了XX功能，解决了YY问题"
```

### 2.2 代码审查规则

- 所有 PR 必须经过至少一人审查
- 审查要点：
  - 代码风格是否符合规范
  - 是否有潜在的安全问题
  - 是否有性能问题
  - 是否有未处理的异常
  - 是否有适当的测试
- 审查未通过需要修改后重新提交

### 2.3 代码质量标准

- 代码注释率 ≥ 30%
- 单元测试覆盖率 ≥ 60%
- 必须通过 ESLint/Prettier 检查
- 必须通过编译检查

## 3. 文档管理规则

### 3.1 文档命名规范

- 使用有意义的英文或中文名称
- 单词之间使用连字符 `-` 分隔
- 示例：`技术详细设计文档.md`、`API接口规范.md`

### 3.2 文档目录规则

| 目录 | 用途 | 包含内容 |
|------|------|----------|
| requirment/ | 需求文档 | PRD、用户故事、需求规格 |
| requirmentReView/ | 需求评审 | 评审记录、评审意见 |
| design/ | 设计文档 | UI设计、交互设计 |
| frontSolu/ | 前端方案 | 前端架构、技术选型 |
| tecSolu/ | 技术方案 | 详细设计、接口规范 |
| sysArchi/ | 系统架构 | 架构设计、数据库设计 |
| testCase/ | 测试用例 | 测试计划、测试用例 |
| projectPlan/ | 项目计划 | 迭代计划、里程碑 |

### 3.3 文档版本管理

- 文档版本号格式：`v主版本号.次版本号.修订号`
- 重大变更：主版本号递增
- 功能增加：次版本号递增
- 文档修正：修订号递增
- 每次更新需要记录变更说明

### 3.4 文档评审规则

- 技术方案需要经过技术评审
- 设计文档需要经过设计评审
- 测试文档需要经过测试评审
- 评审通过后才能进入下一阶段

## 4. 迭代管理规则

### 4.1 迭代目录结构

```
IterationN/                     # N为迭代编号
├── requirment/                # 本迭代需求文档
```

- [ ] 需求文档

- [ ] 敏捷项目管理计划

## 6. 协作流程

### 6.1 日常开发流程

1. **每天开始工作**
   ```bash
   git checkout main
   git pull origin main
   git checkout -b feature/功能名称
   ```

2. **开发过程中**
   - 编写代码
   - 编写测试
   - 本地测试通过
   - 提交代码

3. **每天结束工作**
   ```bash
   git add .
   git commit -m "feat: 今日工作内容"
   git push origin feature/功能名称
   ```

4. **完成后**
   - 创建 Pull Request
   - 等待代码审查
   - 合并到主分支

### 6.2 文档更新流程

1. 在对应的文档目录创建或修改文档
2. 按照文档版本规范更新版本号
3. 提交文档变更
4. 在评审目录添加评审记录（如需要）

## 7. 安全规范

### 7.1 敏感信息管理

- 禁止将敏感信息（密码、密钥、Token）提交到代码仓库
- 使用环境变量或配置中心管理敏感配置
- 敏感信息需要加密存储

### 7.2 访问控制

- 代码仓库访问权限需要严格控制
- 定期清理离职人员的访问权限
- 生产环境密钥需要定期更换

## 8. 备份与恢复

### 8.1 备份策略

- 本地代码定期推送到远程仓库
- 文档随代码一起管理
- 保持本地和远程同步

### 8.2 恢复流程

```bash
# 克隆仓库
git clone git@github.com:whh07333/agent-manage.git

# 切换到对应分支
git checkout main

# 更新到最新版本
git pull origin main
```
