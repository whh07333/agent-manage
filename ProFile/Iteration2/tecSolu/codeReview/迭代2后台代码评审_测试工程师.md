## 一、数据库Schema vs 模型定义一致性检查（核心问题）

根据 `init-db.sql` 实际数据库结构，对比当前 `src/models/` 代码：

### ✅ 已正确对齐
| 模型 | 表名 | 对齐情况 |
|------|------|----------|
| User | `users` | ✅ **已完全对齐**（DEF-001已修复） |
| Project | `projects` | ✅ **已完全对齐**（DEF-002已修复） |
| ProjectAgent | `project_agents` | ✅ **已正确对齐**（模型与DB完全一致） |

### ❌ 存在不一致（按严重程度排序）

---

#### 1. **Task 模型 - 枚举值不匹配 + 命名不一致**
| 问题 | 模型定义 | DB实际 | 严重性 |
|------|----------|---------|--------|
| status枚举值 | 模型期望: `pending, assigned, in_progress, blocked, pending_review, approved, rejected, canceled, completed` | DB约束: `pending, in_progress, completed, blocked, cancelled` (注意拼写是**cancelled**两个L) | 🟥 严重 |
| 字段命名 | `project_id`, `assignee_id`, `parent_id`, `status_remark`, `due_date` (下划线风格) | 模型使用下划线是正确的，但由于模型没有设置 `underscored: true` + 自定义 `timestamps` 映射，整体一致 | ⚠️ 命名风格不统一（模型内混用驼峰和下划线） |

**影响**: 创建任务时，如果status传入 `pending_review` 或 `canceled` 会违反DB CHECK约束，插入失败。

**修复建议**:
1. 修改Task模型status枚举，与DB一致: `['pending', 'in_progress', 'completed', 'blocked', 'cancelled']`
2. 由于模型已经用下划线命名，建议添加 `underscored: true` 配置，避免映射错误

---

#### 2. **AuditLog 模型 - 字段完全不匹配DB schema**
| 模型字段 | DB实际存在 | 问题 | 严重性 |
|----------|-------------|------|--------|
| `actorId` | ❌ 不存在 (`user_id`) | 模型字段名错误 | 🟥 阻塞 |
| `actorType` | ❌ 不存在 | 模型多此字段 | 🟥 阻塞 |
| `actorName` | ❌ 不存在 | 模型多此字段 | 🟥 阻塞 |
| `targetType` | ❌ 不存在 (`resource_type`) | 名称错误 | 🟥 阻塞 |
| `targetId` | ❌ 不存在 (`resource_id`) | 名称错误 | 🟥 阻塞 |
| `changes` | ❌ 不存在 (`content`) | 名称错误 | 🟥 阻塞 |
| `ipAddress` | ✅ 正确映射 (`ip_address`) | - | - |
| `userAgent` | ✅ 正确映射 (`user_agent`) | - | - |
| 缺失 `projectId` 映射 `project_id` | ✅ DB有project_id | 模型有projectId，underscored会正确映射 | ✅ |

DB实际schema (`audit_logs`):
```sql
id, project_id, user_id, action, resource_type, resource_id, content, ip_address, user_agent, created_at
```

**影响**: 任何写入审计日志操作都会报错 `column "actor_id" does not exist`，阻断所有写操作流程（包括创建项目）。

**修复建议**: 完全重构AuditLog模型，对照DB schema:
```typescript
// 正确应该是:
export class AuditLog extends Model {
  public id!: string;
  public projectId!: string | null;
  public userId!: string | null;
  public action!: string;
  public resourceType!: string;
  public resourceId!: string;
  public content!: Record<string, any> | null;
  public ipAddress!: string | null;
  public userAgent!: string | null;
  public readonly createdAt!: Date;
}
// 注意: DB没有updated_at，所以需要: updatedAt: false
```

#### 4. **ApiKey 模型 - 表不存在**
技术方案要求新增 `api_keys` 表，但 `init-db.sql` 中 **没有** 创建此表。模型存在但表不存在。

**影响**: API密钥管理功能完全无法使用，报错表不存在。

**严重性**: 🟥 阻塞（US019整个故事无法使用）

**修复建议**: 在 `init-db.sql` 添加 `api_keys` 表创建语句，按照技术设计文档:
```sql
CREATE TABLE api_keys (
  id UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  agent_id UUID NOT NULL REFERENCES users(id),
  key_prefix VARCHAR(20) NOT NULL,
  key_hash VARCHAR(255) NOT NULL,
  name VARCHAR(255) NOT NULL,
  status VARCHAR(50) NOT NULL DEFAULT 'active' CHECK (status IN ('active', 'revoked')),
  last_used_at TIMESTAMP,
  expires_at TIMESTAMP,
  created_by UUID NOT NULL REFERENCES users(id),
  revoked_at TIMESTAMP,
  created_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP,
  updated_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP
);
```

## 三、编码规范一致性检查

发现的规范问题:

1. **命名风格不统一**:
   - 部分模型使用驼峰命名 (User, Project, ProjectAgent) → 正确（依靠 `underscored: true` 自动映射）
   - 部分模型混用下划线命名 (Task: `project_id`, `assignee_id`; Subscription: `project_id`, `event_types`) → 不符合整体团队驼峰命名模型字段约定
   - DeadLetterEvent: `SubscriptionId` (大写S) → 异常
   - BlockRecord: `TaskId` → 异常

2. **时间戳处理不一致**:
   - User: 依赖 `underscored: true` 自动处理 `createdAt` → `created_at` ✓
   - Project: 同上 ✓
   - Task: 自定义 `timestamps: true, createdAt: "created_at", updatedAt: "updated_at"` → 冗余但正确
   - AuditLog: `updatedAt: false` ✓ （DB没有updated_at）
   - Subscription: 同上 ✓

3. **索引配置不一致**:
   - ProjectAgent 正确添加了索引 ✓
   - DeadLetterEvent 添加了索引 ✓
   - 其他模型大多没有配置索引，虽然DB会在init-db创建，不影响运行
