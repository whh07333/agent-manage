## E2E-IT2-006 详细分析

### 错误栈

```
Unexpected Application Error!
Cannot read properties of undefined (reading 'unassigned')
TypeError: Cannot read properties of undefined (reading 'unassigned')
  at renderItem (src/pages/ProjectList/index.tsx:348:37)
```

### 根因分析

代码在渲染项目卡片统计数据时，直接访问 `project.tasks.unassigned`，没有做空值保护：

```jsx
<div style={{ fontSize: '18px', fontWeight: 'bold' }}>
  {project.tasks.unassigned}  // ❌ 这里 project.tasks 可能是 undefined
</div>
```

当后端API返回的项目数据中 `tasks` 字段不存在或为 `undefined` 时，就会抛出异常，导致整个页面渲染失败。

### 影响范围

一共有三处都存在这个问题：
- `project.tasks.unassigned` (第348行)
- `project.tasks.inProgress` (第355行)  
- `project.tasks.completed` (第362行)

### 修复建议

使用可选链操作符 `?.` 并提供默认值 `0`：

```jsx
<div style={{ fontSize: '18px', fontWeight: 'bold' }}>
  {project.tasks?.unassigned ?? 0}
</div>
```

## E2E-IT2-009 managerId 类型不匹配，后端期望 UUID 格式默认值 "1" 不合法

### 缺陷基本信息

| 字段 | 内容 |
|------|------|
| **缺陷ID** | E2E-IT2-009 |
| **模块** | 项目管理 -> 创建项目 |
| **严重等级** | 🔴 阻塞级别（创建流程无法完成） |
| **发现时间** | 2026-03-23 |
| **状态** | ✅ 已修复 |

### 缺陷描述

创建项目时，`managerId` 默认值设置为 `"1"`，但后端数据库字段类型是 UUID，无法解析 `"1"` 这个值，导致SQL执行错误：

```
{"code":500,"msg":"invalid input syntax for type uuid: \"1\"","data":null}
```

### 现状分析

前端：
- 默认值设置为 `"1"`，方便测试填写
- 但数据库字段类型是 UUID，`"1"` 不是合法的 UUID 格式

### 修复建议

方案选择其一即可：

1. **方案A（推荐）**: 修改数据库字段类型为 `integer`（自增ID），因为是管理员ID=1，整数合理
2. **方案B**: 给默认值一个合法的 UUID，比如 `"00000000-0000-0000-0000-000000000001"`

## E2E-IT2-010 默认managerId对应的用户不存在，违反外键约束，创建失败

### 缺陷基本信息

| 字段 | 内容 |
|------|------|
| **缺陷ID** | E2E-IT2-010 |
| **模块** | 项目管理 -> 创建项目 |
| **严重等级** | 🔴 阻塞级别（创建流程无法完成） |
| **发现时间** | 2026-03-23 |
| **状态** | ✅ 已修复 |

### 缺陷描述

创建项目时，使用默认 `managerId`，但该ID对应的用户在 `users` 表中不存在，违反外键约束，创建失败：

```
{"code":500,"msg":"insert or update on table \"projects\" violates foreign key constraint \"projects_manager_id_fkey\"","data":null}
```

### 根因分析

- 数据库中 `projects.manager_id` 有外键约束引用 `users.id`
- 前端给 `manager_id` 设置了默认值（无论是 `"1"` 还是 UUID `"00000000-..."`），但对应ID的用户不存在
- 导致插入失败

### 修复方案

需要在测试环境数据库中确保至少存在一个用户，ID与默认值匹配。可以：

1. 在 `init-db.sql` 中插入一个默认管理员用户，ID=1（如果使用integer）
2. 或者插入一个默认管理员用户，ID使用对应的合法UUID
3. 或者去掉非必要的外键约束（如果业务上不强制需要）
